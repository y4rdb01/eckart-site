Code Commenting Guidelines

The following guidelines are due to Bjarne Stroustrup and appear on page 105 (2nd edition) of his book The C++ Programming Language. Comments in [emphasis and brackets] are my own.

Stroustrup's preference is for:
  1. a comment for each source file stating what the declarations in it have in common, references to manuals, general hints for maintenance, etc.;
  2. a comment for each class [module of package] or template [generic];
  3. a comment for each nontrivial function stating its purpose, the algorithm used (unless it is obvious), and maybe something about the assumptions it makes about its environment;
  4. a comment for each global variable;
  5. a few comments in places where the code is nonobvious and/or nonportable; and
  6. very little else.

I would also add to the above that comments should bridge the gap between problem and code. That is, comments should help the reader of the program understand how the code solves the problem.
