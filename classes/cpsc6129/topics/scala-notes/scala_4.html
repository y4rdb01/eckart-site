Scala Programming Language
Object-Oriented Scala

  1. Overview
    • Though Scala has full support for functions and functional programming, functions are not top-level constructs in Scala
    • While we are able to enter unenclosed functions into the Scala REPL, when we are writing program code for compilation, functions must sit inside of a top-level construct
    • Scala supports 4 top-level constructs:
      1. Objects
      2. Classes
      3. Case classes
      4. Traits

  2. Objects
    • Objects have been given previous treatment
    • The object keyword creates a type and a single instance of it
    • Common Scala practice generally utilizes companion objects, which hold static class behavior, while instance-specific behavior is encapsulated in the classes themselves
    • Related classes and objects can have the same name, which is a common practice in Scala

  3. Classes
    • Classes have been given previous treatment
    • Important points to reiterate include:
      1. Constructor parameters are integrated into the class definition signatures
      2. Common practice of "companion objects," which hold static code associated with instance-specific class code
    • Inheritence works similar to Java, in that classes can be extended to form subclasses
    • Abstract classes are supported
    • Traits and mixins can also be combined to form new classes (see below)

  4. Case Classes
    • Case classes care immutable data objects that depend exclusively on their constructor arguments
    • This is a functional concept, and allows pattern matching to be brought to objects, and implicit equality comparisons to be defined
    • Case classes can be further investigated here

  5. Traits & Mixins
    • Similar to interfaces in Java
    • Collections of attributes and behaviors that can be extended by, or mixed in to, your classes
    • Creating new classes by extending other classes or traits, and adding in additional characteristics by selecting additional traits, is referred to as mixin composition. 
      
	/** Define a few traits. */

	trait SmartPhone {
	    val make: String
	    val screenSize: Double
	}

	trait Android {
	    val androidVersion: Double
	}

	// Use a single trait with keyword 'extends'.
	class Samsung extends SmartPhone {
	    val make = "Samsung"
	    val screenSize = 4.8
	}

	// Use several traits with additional keyword 'with' (mixin composition).
	class Samsung extends SmartPhone with Android {
	    val make = "Samsung"
	    val screenSize = 4.8
	    val androidVersion = 5.0
	}
  6. Getters & Setters
    • Unlike Java, getters and setters are not really required, at least not in the default case
    • The division of vars and vals makes controlling access to, and mutation of, attributes more defined
    • vals are not mutable, and so attempting to access and change them is not possible
    • vars, on the other hand, are mutable, and so direct access to change them is reasonable
    • The default visibility in Scala is public, as it is reasoned that this is the most popular visibility, and so this strategy of expecting direct access to vars and vals is reasonable in the general case
    • Of course, Scala provides different visibility scopes, and so classes and methods can be implemented according to requirements, but as a default, this previously outlined practice works quite well, and cuts down on unnecessary and repetitive typing (excessive use of final, public, and constant implementations of setVar, getVar, getVal, etc.)
    • For more on visibility scopes in Scala, see here

  7. The Uniform Access Principle
    • The Uniform Access Policy states that, "[a]ll services offered by a module should be available through a uniform notation, which does not betray whether they are implemented through storage or through computation." [here]
    • Simply put, the syntax used to access some part of a class should be the same whether an attribute or a method
    • In Java, this does not hold: 
      
	// Create an instance of MyClass.
	MyClass a = new MyClass();

	// Access myVar and myMethod.
	int i = a.myVar;
	String s = a.myMethod();
    • As you can see, the empty parentheses at the end of the method call are not present at the end of the method call, != Uniform Access Principle
    • However, thanks to its flexible syntax, this is true of Scala: 
      
	// Create an instance of MyClass.	
	val a: MyClass = new MyClass
	
	// Access myVar and myMethod.
	val i = a.myVar
	val s = a.myMethod
    • The imprints of both of these calls match, and therefor stasify the Uniform Access Principle
    • While not a practical requirement of an effective language, that Scala satsifies this principle is a testament to its academic and functional roots
